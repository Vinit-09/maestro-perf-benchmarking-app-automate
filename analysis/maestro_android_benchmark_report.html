01 · The Question

Why this benchmark exists

BrowserStack runs Maestro tests across thousands of Android devices. Customers ask one question over and over: "How much slower is the cloud than my local laptop?" We needed a defensible number.

The hypothesis

BrowserStack cloud Android Maestro should run within 1.1× of an equivalent local run. That's the internal SLA target — the number that determines whether the cloud experience is "indistinguishable from local" for a customer.

What's in scope

  • Samsung Galaxy S24 / Android 14 on cloud, OnePlus 9R / Android 14 on local
  • A Maestro flow sized to ~1153 s P90 (spec target = 5/13 smoke baseline)
  • Same-region cloud devices (ap-south-1 for India operators)
  • Per-session, per-testcase and per-rep latency at P50, P75, P90, P95, P100

What we measured

Total session time is a chain of phases that BrowserStack and the customer both touch. Each card below names the phase and explains every step inside in plain language.

Phase 1

Pre-Maestro

Everything that happens before Maestro itself even starts running.

Device probe
Check that the Android device is connected and reachable. Like asking "is the phone there and turned on?" before doing anything else.
App install
Install WikipediaSample.apk onto the device. Same idea as installing from the Play Store, just pushed from a host (local) or BrowserStack's server (cloud).
Phase 2

Maestro startup

The Maestro test engine wakes up and gets ready to drive the device.

JVM boot (maestro_init)
Maestro is written in Java/Kotlin. The Java engine has to start up first — like turning the key in a car before you can drive. About 1.5 seconds locally.
Driver setup
A small helper app (the "driver") runs on the Android device and is what actually taps, types, and reads the screen. Maestro opens a connection to that driver before any test can happen.
Device info
Maestro asks the phone "what model are you, what's your screen size, what version of Android?" — quick metadata fetch.
Phase 3

Flow execution

The actual test work — opening the app and doing repeated user actions.

launchApp
Open Wikipedia Alpha on the device, just like a user tapping the icon on their home screen.
Tap on search
Tap the search field at the top of the app, the same way a user would touch it to start typing.
The repeat block (×100)
Three actions done 100 times in a row: clear the search box → type "BrowserStack" → confirm the result "Software company based in India" shows up. This stresses the system. On cloud, this block is split into two 50-rep testcases (test_a + test_b) to fit BrowserStack's per-testcase time cap.
Phase 4

Post-Maestro

Tidying up after the test ends.

App uninstall + cleanup
Remove Wikipedia Alpha from the device so the next test starts clean. Closes the driver-app connection.
02 · The Setup

Same app. Same flow. Two environments.

To compare apples to apples, we used a single Android APK — WikipediaSample.apk (Wikipedia Alpha, org.wikipedia.alpha) — and ran an identical Maestro flow locally and on BrowserStack's cloud.

WikipediaSample

The official open-source Wikipedia Alpha APK. Stable selectors (org.wikipedia.alpha:id/search_container) so the flow is selector-deterministic. Same .apk uploaded to BrowserStack as installed locally.

Version 2.5.194-alpha-2017-05-30. Cloud uploads the APK to BS App Automate; local sideloads via adb install.

The flow

One workload, two homes. Identical commands; cloud splits into two YAMLs to fit the per-tc cap.

- launchApp
- tapOn:
    id: "org.wikipedia.alpha:id/search_container"
- repeat:
    times: 50  # ×2 yamls on cloud, ×1 100-rep yaml local
    commands:
      - eraseText
      - inputText: "BrowserStack"
      - assertVisible: "Software company based in India"

Sizing the workload to ~1153 s P90

The spec target is 1153 s P90, derived from the 5/13 single-session smoke build (build 6cbf79e4af4872135a2eb0373c6498e28c61cbfb, session_total = 1153 s on Samsung Galaxy S24-14.0). At 100-session scale, cloud P50 landed at 1139.0 s — within 2 % of spec.

per-tc workload ≈ 50 reps × ~10.6 s/rep ≈ 530 s
per-session overhead (launchApp + tap × 2 tcs) ≈ ~85 s
session_total ≈ 530 + 530 + 85 = ~1145 s (smoke validated, scaled to 100 sessions)
03 · The Journey

What it took to get a clean number

Benchmarks succeed or fail in the plumbing. Four real bugs / engineering decisions sat between us and trustworthy data — most discovered during the run itself.

1 — The dual-yaml engineering decision (gotcha #3 mitigation)

BrowserStack Maestro v2 enforces a ~900 s per-testcase execution-time cap. A naive single 100-rep yaml would push tc.duration past the cap and BS would silently kill the test mid-flow — verified against builds at 148-rep, 150-rep, and 132-rep scale, all killed at ~940–975 s.

What this is about: BrowserStack quietly caps how long one Maestro test can run (around 15 minutes per testcase, not per session). A test running 100 reps of search-and-assert sometimes pushes past that cap — and BS kills it with no error message, no stack trace, no warning. Just session marked failed with metadata showing zero flows completed.

The fix: Split the 100 reps across two identical testcases (test_a + test_b, 50 reps each) in the same session. BS gives each testcase its own ~900 s budget. Total work per session stays the same; each tc finishes comfortably under the cap.

This run validates that mitigation at scale. Across 97 passed cloud sessions (194 testcase runs), test_a P90 = 532 s, test_b P90 = 531 s — well under the 870 s inspect threshold. Zero cap-kill violations.

2 — The sick device unit (gotcha #16): 3 sessions lost on the same hostname

3 of 100 sessions errored at exactly ~322 s with byte-identical fingerprints. All three landed on the same physical Samsung Galaxy S24 unit (hostname RZCX60PWTST). BS re-allocated the same sick device three times instead of pulling it from the pool.

What this is about: Cloud test pools are heterogeneous — BrowserStack runs your test on whichever physical device happens to be free. Most of the pool is healthy, but every once in a while you get a sick device: maybe a background process is hogging CPU, maybe the OS is mid-update, maybe Bluetooth has flooded the dmesg log. Today we hit one such sick device three times in a row.

The fingerprint: session lasts ~320 s, both testcases report duration 0 with empty stacktrace, Maestro log stops at "Selected device RZCX60PWTST" before launchApp, generic BS error message "Could not start a session. Please try to run the test again." Different from gotcha #1 (Instrumentation stalled at 36–65 s) — this one is a ~125 s pre-launchApp watchdog kill, documented as gotcha #16.

Pool-allocation observation: 8 passed sessions in the same build hit 8 distinct other hostnames. BS spreads sessions widely, but kept re-picking RZCX60PWTST for 3 attempts despite repeated failure — there is no API today to blacklist a known-bad hostname mid-build.

3 — The runner script crashed on a transient curl error

Mid-build, cloud_run_android.sh's 30-second poll loop got a curl: (56) Recv failure: Connection reset by peer. The script's JSON parse threw JSONDecodeError, $BSTATUS resolved to "?", and the loop broke — writing a 1-byte build JSON and empty sessions.txt.

What happened: The runner's curl to BS's status endpoint hit a transient TCP reset (Recv failure). The script doesn't retry on transient errors — it just tries to parse an empty response, JSON-decode fails, $BSTATUS becomes the literal string ?, and the poll-loop exit condition != "running" matches that string. Loop breaks. Script writes corrupted state to disk and exits 0.

The recovery: BS doesn't know the local runner died. The build continued cooking on BS's side. We re-queried the live build state via the API, overwrote the corrupt 8ed0b0edc7e137…json with the real build state, and regenerated sessions.txt from the live data. No data loss — just operator overhead.

The fix (recommended follow-up): Add a curl-retry guard around the poll-loop request. On transient failure, log + sleep + retry instead of treating the corrupt response as terminal.

4 — The BS dashboard counts testcases, not sessions

Mid-build the BS App Automate dashboard showed "48 PASSED" with a counter ticking at "0 7h 35m 52s" — appearing stalled. The 48 is testcase count (24 sessions × 2 tcs); the runner counts sessions. The 7h 35m is cumulative session-time on the device pool, not remaining wall-time.

What this is about: BrowserStack's web dashboard surfaces aggregate counters that look like sessions but are actually testcase-scoped. For a dual-yaml workload, "48 PASSED" means 24 sessions completed × 2 testcases each. The cumulative-duration counter (7h 35m 52s on screen) is the total billable wall-clock the pool has spent on this build — useful for billing math, misleading as a progress signal.

Why it matters: An operator watching the dashboard would mistake healthy progress for a stalled build. The runner's poll output ("running=76 passed=24") is the authoritative session-level count. This is a UI/data-model mismatch worth flagging to the BS team.

Three things this mapping makes clear

Cloud has overhead local can't have

Three cloud phases (start_sessions, app_dl, test_dl) are pure cloud-only work — the cost of BrowserStack owning the device pool, blob storage, and signing infrastructure. Local doesn't have these because the customer's host already has the files.

Per-tc data is REST-only, not BQ

For dual-yaml workloads, the per-testcase breakdown is the most useful diagnostic — but BigQuery aggregates it away. The per-tc timings in this report come from N+1 REST calls (one build endpoint + N session-detail endpoints). Recommendation #3 (surface per-tc in BQ) addresses this.

Cloud wins on flow execution, loses on install

Cloud app_install is ~4.6 s slower at P90 (BS install pipeline overhead). But cloud's flow execution beats local because the S24's newer SoC outpaces the OnePlus 9R on raw tap/type/assert throughput. Net result: cloud session_total wins by 100 s at P90.

08 · Recommendations

Why cloud is faster — and how to extend the lead

Cloud Samsung Galaxy S24 beats wired OnePlus 9R at session_total P90. Two reasons cloud wins, three places BrowserStack can squeeze more headroom.

Why cloud wins (the per-rep math)

1. Newer SoC, faster flow execution

The Samsung Galaxy S24's SoC outpaces the OnePlus 9R (Snapdragon 870) on raw per-rep throughput. Cloud per-tc P50 = 528.0 s per 50-rep testcase, or ~10.56 s/rep. Local per-rep P50 = 8.58 s/rep. Hardware deltas compound across 100 reps.

2. Cloud avoids local's USB-bridge overhead

Local Maestro runs over an adb connection via USB. Every tap, type, and assert command serializes through that bridge, adding microseconds-to-milliseconds per command. Cloud BS-managed devices have the driver pre-installed on the same physical pipeline as the OS, eliminating that bridge.

How BrowserStack can extend the lead

High impact

1 — Reduce app_install overhead via APK warm-caching

Cloud app_install P90 is 6,564 ms — 4.6 s slower than local at P90. For benchmark workloads where the same APK runs 100+ times in a row, that's pure repeated work.

What this is about: Every cloud session re-installs the APK from BS's blob storage, even though the same APK was just installed seconds earlier. It's like a barista re-grinding their beans for every cup instead of using the same grinder all morning.

Why it matters: A customer running 100 benchmark sessions pays this ~6.5 s tax 100 times — that's over 10 minutes of pure waste, with no benefit. Even a 50% reduction (cache the resolved APK URL + content hash for a "warm" window) recovers ~5 min on a 100-session run.

Fix: Detect repeated APK uploads via content hash; serve from a host-local cache instead of re-fetching from blob storage. Optional opt-in flag for benchmark mode.

High impact

2 — Detect and quarantine sick devices (gotcha #16)

3 of 100 sessions in this build landed on the same physical Samsung Galaxy S24 unit (hostname RZCX60PWTST) and all 3 errored identically at ~322 s. BS keeps re-allocating the same sick device.

What this is about: When a device fails the ~125 s no-flow-progress watchdog twice in quick succession on the same hostname, BS's pool allocator should treat that hostname as suspect and exclude it from further allocations in the same build (or platform-wide for some cool-off period).

Why it matters: Without quarantine, the same sick device can fail an unbounded number of sessions in a single build. We saw 3 in 100 (3 % impact); a sicker pool could see 10+. For benchmark workloads where statistical density matters, this directly reduces sample size.

Fix: Mid-build hostname blacklisting (BS-internal allocator-level), surfaced as an API hint to operators ("session X retried on a different hostname after Y").

Medium impact

3 — Surface per-testcase timings in BigQuery

For dual-yaml workloads, per-tc duration is the single most useful diagnostic. Today it requires N REST calls — one per session — to assemble. BigQuery aggregates testcase data away.

What this is about: Maestro keeps per-testcase logs that BS already captures (the per-tc duration field is in the REST /sessions/<sid> endpoint). BQ flattens this into the session-level duration and a single JSON-string test_status counts blob.

Why it matters: Operators running dual-yaml workloads can't query "show me builds where test_a P90 spiked relative to test_b" or "show me sessions where test_b errored while test_a passed". Today that's a 100-call REST iteration.

Fix: Add per-tc rows to BQ (one row per session per tc) with at minimum: tc_class, tc_duration, tc_status, tc_start_offset_ms. Diagnostic-only; no perf impact.

Bottom line

Cloud already wins on raw session_total P90. The per-rep workload (~92 % of session_total) is hardware-bound — and BS's S24 hardware wins. The reachable budget is the app_install pipeline; recommendation #1 alone recovers ~5 s P90 and pushes cloud's lead from 100 s to ~105 s without any change to the workload.

09 · Methodology

How we measured

For reproducibility and audit. Skip if you trust the numbers.

Local measurement stack
  • Host: macOS 25.4, ADB platform-tools current.
  • Device: OnePlus 9R (Android 14), USB-tethered.
  • Maestro CLI: 1.39.13 (stable release).
  • Test runner: Maestro driver pre-installed once; reused across all sessions.
  • Per-iter cycle: uninstall app → install app → maestro test → uninstall app.
  • Phase parsing: Maestro debug log parsed by parse_maestro_log.py for step-level timings; bash wrapper records phase boundaries via now_ms wall-clock.
Cloud measurement stack
  • Build trigger: One BrowserStack Maestro build with 100 device entries (Samsung Galaxy S24-14.0, ap-south-1).
  • Workload split: 2 yamls in execute array (test_a.yaml, test_b.yaml), 50 reps each — gotcha #3 split-yaml mitigation.
  • Concurrency: ~25 sessions running in parallel waves (BS plan parallel cap).
  • Status polling: BS REST API every 30 s until terminal.
  • Per-session BQ metrics: Pulled from browserstack-production.app_automate.app_automate_test_sessions_partitioned ~50 min after build completion, filtered with JSON_VALUE(test_status, '$.error') = '0' to exclude the 3 gotcha-#16 errored sessions.
  • Per-tc timings: N+1 REST calls — 1 build endpoint to list session ids, N session-detail endpoints (/maestro/v2/builds/<bid>/sessions/<sid>) to extract testcases.data[].testcases[].duration.
Percentile method

Linear-interpolation percentile (numpy default). For repeating loop-body steps, N = sessions × reps_per_session = 900 on local.

Cloud aggregates include all 97 passed sessions; the 3 errored sessions on hostname RZCX60PWTST (gotcha #16) are excluded by BQ filter. One outlier passed session at 2054 s appears at P100 only.

Sample sizes & confidence
  • Local: 9 sessions, narrow spread. Confidence high.
  • Cloud: 97 sessions (passed only). Range: 1036–2054 s. P50–P95 in a tight 1139–1162 s band; P100 is the single outlier.
  • Sample size for cloud P90: at N=97, the P90 estimate has ±1 rank uncertainty. The 1155.8 s figure is robust.
