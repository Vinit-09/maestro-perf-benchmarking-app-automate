BrowserStack Maestro v2 has two distinct failure modes that don't appear in any docs.

A full day of probing the cloud Android benchmark pipeline surfaced two platform behaviors that aren't documented anywhere on BrowserStack's side and that both produce session-level failed with empty stacktraces. Two findings

TL;DR — both failures look identical at the API level. They aren't.

Both produce session.status: "failed", empty stacktrace, maestro_status: "UNKNOWN", and completed_flows: 0/0. The difference shows up only when you bucket by tc.duration / session.duration and look at the maestro log tail.

Precision notes for leadership

What this report claims, and what it does not.

  1. The platform cap is on Maestro test execution time (tc.duration) at ~900 s (observed kill band 883–914 s). session.duration at kill looks like 940–975 s because BS adds ~60 s of pre/post-test overhead. The "900 s" number is real; it applies to the tc.duration field, not session.duration.
  2. The cap is independent of the idleTimeout build-payload key. Direct comparison: build 1687bb73…832e5 (sent idleTimeout: 900) was killed at tc.dur ~889 s; build f3c2c50a…d46 (did not send idleTimeout) was killed at tc.dur ~883 s. The cap fires at the same tc.dur regardless of the payload key.
  3. We did not verify what idleTimeout actually controls on Maestro v2. BS docs describe it as a no-activity timer, but all our flows had sub-second gaps between commands, so any idle behavior was never exercised. This is an open question, not a finding.
  4. We did not verify that "splitting the flow across multiple .yaml files in the test_suite zip gives each file its own ~900 s budget." It is a plausible workaround but should be tested before being relied on for a production pipeline.
Finding 1 — platform-wide

BS-side cap at tc.duration ≈ 900 s

BS Maestro v2 kills any single Maestro test once Maestro test execution time (tc.duration) reaches ~880–915 s. Corresponding session.duration at kill is ~940–975 s (BS adds ~60 s of pre/post-test overhead). The cap fires independently of the idleTimeout build-payload key — verified in builds that explicitly set idleTimeout: 900 as well as builds that did not. Verified across two device pools (OnePlus 12R-14.0, Galaxy S25-15.0). Setting sessionTimeout: 1800 was silently dropped and did nothing.

Finding 2 — device-pool-specific (today's data)

OnePlus 12R-14.0 cold-launch instrumentation stalls

On 2026-05-11, OnePlus 12R-14.0 in the ap-south pool stalled on cold-launchApp in every probe (5 / 5) and in ~47 % of a 100-session fleet run. Galaxy S25-15.0 in the same pool, same time window, had no stalls (3 / 3 probes). Sample sizes are small for Galaxy and the stall appears time-correlated, so we cannot claim Galaxy is stall-free in general — only that the OnePlus pool is presently degraded relative to the Galaxy pool.

Symptoms: session.status: "error", duration 36–65 s, stacktrace: "Instrumentation stalled", maestro log stops at Launch app … RUNNING, device log 0 bytes.

Finding 1 · platform-wide

The cap fires at tc.duration ≈ 900 s — and it's not idleTimeout

Two BrowserStack concepts are easy to confuse. This finding is about the observed cap on Maestro test execution time — not about the idleTimeout build-payload key. We tested both with and without that key; the cap fired the same way in both cases. The idleTimeout capability's actual semantics on Maestro v2 remain unverified by our experiments.

What we have NOT tested — be aware of these gaps

  • Whether idleTimeout does what its name suggests on Maestro v2. BS docs describe idleTimeout as "maximum time between consecutive commands" for Espresso/XCUITest. All our flows had <1 s between commands, so the no-activity timer (if any) never fired. We cannot say whether a flow with intentional multi-second idle periods would behave differently. To test: run a flow with extendedWaitUntil or a delay of 120+ s, with idleTimeout: 60.
  • Whether the tc.duration ≈ 900 s cap is the documented idleTimeout mechanism enforced as wall-time, or a separate platform-level execution-time cap, or something else. Our data shows behavior, not the underlying mechanism. The mechanism's identity matters because it determines whether any future BS payload key could lift it.
  • Whether splitting a flow into multiple .yaml files inside the test_suite zip gives each file its own ~900-s budget. Plausible based on how BS reports tests, but not confirmed. Need to run a test_suite with two .yaml files of ~500-s each and observe whether the second one is allowed to start and complete.
  • Whether the cap applies identically on iOS Maestro v2. We have not run the same probe set on the iOS pipeline.

No error message anywhere. The only way to know how far it actually got is to grep the maestro log:

Mitigation

Keep a single Maestro .yaml test under tc.duration ≈ 870 s of expected execution time (≈ 930 s of session.duration with typical BS overhead). No build-payload parameter we've tested lifts the cap; idleTimeout and sessionTimeout do not. If the workload genuinely requires more, the only confirmed path is to contact BS support and request a higher cap for this account. Flow-splitting into multiple .yaml files inside the test_suite zip is plausible but unverified — it should be tested before relied on.

Finding 2 · device-pool-specific

OnePlus 12R-14.0 cold-launch instrumentation stalls

On 2026-05-11, the OnePlus 12R-14.0 ap-south pool exhibited a high rate of cold-launchApp stalls for our Wikipedia Alpha APK. The session never gets past the very first Maestro command. What to do

Recommendations

The benchmark requirement is 100 reps of identical work. Both findings are survivable for that target with the right operational posture.

High impact

1. Cap your single Maestro test at tc.duration ≤ ~870 s.

100 reps × ~8.1 s/iter ≈ 738–873 s of tc.duration across the device pools we tested — fits with margin. If a future workload requires more, the only safe path until BS confirms otherwise is to request a higher cap from BS support; the "split into multiple .yaml files" trick is plausible but untested.

High impact

2. Add session-level retry to cloud_run_android.sh.

For OnePlus 12R-14.0 (or any pool with stall noise), after a build terminates, identify error-status sessions with duration < 90 s and re-fire them as a fresh build. Repeat until N clean passes. This treats pool heterogeneity as a known noise source without changing device or flow.

Med impact

3. Footnote OnePlus 12R-14.0 results in the final report.

State the observed pool-stall rate as a measurement caveat — OnePlus 12R-14.0 numbers reflect BS pool health on 2026-05-11, not just framework behavior. Galaxy S25-15.0 numbers don't need this footnote.

Med impact

4. File a BS support ticket with the hostnames.

Provide b33588eb, 5cbb0293, 56c5aa31, d19684da and the Instrumentation stalled evidence. Long lead time, but the right durable fix for the OnePlus 12R-14.0 pool.

Low impact

5. Skip unknown BS payload keys — verify them via input_capabilities.

BS silently drops unrecognized build-payload keys (sessionTimeout proven). Inspect GET /maestro/v2/builds/<id>'s input_capabilities after triggering a build — if a key you passed isn't there, it wasn't honored. Trust this over BS docs.

Low impact

6. Don't try to stop running Maestro v2 builds via API.

All DELETE/POST/PUT/PATCH variants return 422 or 404. Use the BS App Automate web dashboard instead — that works, verified today.
