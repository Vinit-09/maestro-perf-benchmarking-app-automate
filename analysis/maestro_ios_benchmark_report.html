01 · The Question

Why this benchmark exists

BrowserStack runs Maestro tests across thousands of iOS devices. Customers ask one question over and over: "How much slower is the cloud than my local laptop?" We needed a defensible number.

The hypothesis

BrowserStack cloud iOS Maestro should run within 1.1× of an equivalent local run on the same model device. That's our internal SLA target — the number that determines whether the cloud experience is "indistinguishable from local" for a customer.

What's in scope

  • iPhone 17 / iOS 26.x on both sides
  • A Maestro flow sized to ~733 s P90 (per spec)
  • Same-region cloud devices (ap-south-1 for an India operator)
  • Per-session and per-rep latency at P50, P75, P90, P95, P100

What we measured

The total session time isn't a single number — it's a chain of phases that BrowserStack and the customer both touch. Each card below names the phase and explains every step inside it in plain language.

Phase 1

Pre-Maestro

Everything that happens before Maestro itself even starts running.

Device probe
Check that the iPhone is connected and reachable. Like asking "is the phone there and turned on?" before doing anything else.
App install
Install the HelloBench app onto the iPhone. Same idea as installing an app from the App Store, just pushed from a Mac (local) or BrowserStack's server (cloud).
Phase 2

Maestro startup

The Maestro test engine wakes up and gets ready to drive the phone.

JVM boot (maestro_init)
Maestro is written in Java. When you launch it, the Java engine has to start up first — like turning the key in a car before you can drive. About 1 second.
Driver setup
A small helper app (the "driver") runs on the iPhone and is what actually taps, types, and reads the screen. Maestro has to open a connection to that driver before any test can happen.
Device info
Maestro asks the iPhone "what model are you, what's your screen size, what version of iOS?" — quick metadata fetch so it knows what it's working with.
Phase 3

Flow execution

The actual test work — opening the app and doing repeated user actions.

launchApp
Open (start) HelloBench on the device, just like a user tapping the icon on their home screen.
tapOn searchField
Tap on the search box at the top of the app, the same way a user would touch it to start typing.
The repeat block (115×)
The same three actions, done 115 times in a row: clear the search box → type the word "Browser" → confirm the result "BrowserStack" shows up. This is what stresses the system.
Phase 4

Post-Maestro

Tidying up after the test ends.

App uninstall + cleanup
Remove HelloBench from the iPhone — like deleting an app you installed temporarily — so the next test starts clean. Also closes the helper-app connection.
02 · The Setup

Same app. Same flow. Two environments.

To compare apples to apples, we built a single iOS app — HelloBench — and ran an identical Maestro flow on local hardware and on BrowserStack's cloud.

HelloBench

A SwiftUI app with a search field over a 30-company list. Stable accessibility ids (searchField, item_BrowserStack) so the flow is selector-deterministic.

Built in Xcode 26.4.1, signed with a free Personal Team (***). Same .ipa uploaded to BrowserStack as installed locally.

The flow

One YAML, two homes. Identical commands; only rep count differs (2 for smoke, 115 for benchmark).

- launchApp
- tapOn:
    id: "searchField"
- repeat:
    times: 115
    commands:
      - eraseText
      - inputText: "Browser"
      - assertVisible:
          id: "item_BrowserStack"

Sizing the workload to ~733 s P90

The spec target is 733 s P90 — long enough to stress the harness, short enough to fit a parallel BrowserStack quota window. Calibration math:

per-rep cost ≈ 5.86 s (smoke)
per-session overhead ≈ 12.8 s
N = (733 - 12.8) / 5.86 ≈ 122.9 → 115 reps chosen for first real run
03 · The Journey

What it took to get a clean number

Benchmarks succeed or fail in the plumbing. Three real bugs sat between us and trustworthy data — none of which were on the spec sheet.

1. Maestro 2.5.0's iOS real-device path was broken upstream

Stock Maestro CLI throws NotImplementedError on launchApp for iOS 17+ physical devices. Confirmed against open issue #2907 and PR #3100.

What is "out-of-the-box Maestro"? Maestro is a free, open-source tool that lets you write simple YAML scripts to automate apps on iOS and Android. "Out-of-the-box" means the version you get when you run the official one-line installer from their website — version 2.5.0 was the latest stable release when we started. This is the version every customer would be using by default.

Why couldn't it start an app on a real iPhone? Maestro supports two kinds of iOS targets: simulators (a fake iPhone running on your Mac) and real iPhones plugged in over USB. On Apple's modern iPhones (iOS 17 and newer, including our iPhone 17 on iOS 26), Apple replaced the old way of installing and launching apps with a new tool called devicectl. Maestro 2.5.0 was supposed to use devicectl for real-device support — but the developers had only put placeholder code in there ("TODO: not yet implemented"). The moment our test tried to launch HelloBench on the iPhone, Maestro crashed with a "this function doesn't exist yet" error.

What is a "community-patched version" and why did we build one? Because Maestro is open-source, anyone can submit improvements via a pull request (PR) — a proposal to merge new code into the official project. We found PR #2856 from a developer named omnarayan who had already written the missing code: it actually calls devicectl to launch the app, makes the on-device helper listen on the right network address, and routes the right port number through. The catch is that the PR was still pending review by the Maestro maintainers — not yet officially released. So we downloaded the source code, applied that PR's changes, plus one extra one-line fix we found ourselves, and compiled our own working build of Maestro. Without this step, no measurement on a real iPhone would have been possible at all.

The takeaway: a customer trying to benchmark on a brand-new iPhone today using stock Maestro would simply fail at step one. The official tooling has not caught up to Apple's latest devices. Local iOS benchmarking is currently a "build it yourself" exercise — something worth highlighting to the broader Maestro and BrowserStack communities.

Fix: Rebuilt Maestro CLI from PR #2856 (omnarayan fork) which adds the missing devicectl-based launch logic, fixes the on-device XCTest runner's 0.0.0.0 bind, and patches a one-line port hardcode in TestCommand.kt:496.

2. iproxy USB tunnel stalls after ~7 minutes of traffic

Our 115-rep flow takes ~11 minutes. iproxy (libimobiledevice) silently drops the TCP keep-alive at ~rep 72 every time. The XCTest runner stays alive — the bridge dies.

The cable connection between the Mac and iPhone uses a small bridge program. After about 7 minutes of constant chatter, that bridge silently dies — like a phone call dropping mid-conversation. The phone is still listening, but the line is dead, so every test failed at the same point. We swapped the bridge for a more reliable one (pymobiledevice3) and the next run completed cleanly.

Fix: Replaced iproxy with pymobiledevice3 usbmux forward. First validation run completed all 115 reps cleanly in 679 s.

3. BrowserStack BigQuery doesn't expose Maestro debug-log step timings

Cloud BQ surfaces session-level phases (firecmd_ms, app_install_ms, test_install_ms, etc.) but not per-step (eraseText, inputText, assertVisible) durations. Limits comparability with local.

For local runs we know exactly how long each tap, type, and assertion took. For cloud runs we only get the big-picture totals — BrowserStack doesn't capture or store the fine-grained step timings, even though Maestro produces them. So our cloud table can fill in the totals but not the inner-loop detail; this is something BrowserStack could add to give customers better visibility.

Workaround: Reports use the local report template; cloud-only rows show what's available, missing rows stay blank.

Side-quest: 2 cloud sessions stuck "running" for 2+ hours

Out of 100 device entries, 2 sessions never transitioned to a terminal state. BS's idleTimeout (1800 s) didn't fire. We accepted the dataset at 98 sessions and the percentile shapes are stable at this sample size.

Three things this mapping makes clear

Cloud has overhead local can't have

Three cloud phases (start_sessions, app_dl, test_dl) are pure cloud-only work — the cost of BrowserStack owning the device pool, blob storage, and signing infrastructure. Local doesn't have these because the customer's Mac already has the files.

Cloud is coarser inside the loop

BS exposes one number (duration) for everything local breaks into 8 sub-phases. This is the BigQuery schema gap that recommendation #3 addresses — it's not a perf issue, it's a visibility issue. Customers can't diagnose what's slow.

firecmd_ms is the swing variable

It's the single biggest "what's happening here?" bucket — and where the cloud-vs-local gap mostly lives. Outlier sessions in this bucket blow up to 90+ seconds. Recommendation #1 (reduce firecmd outliers) is the highest-leverage fix.

What the data tells us

✓ Cloud beats local on app_install

⚠ Pre-flight overhead is the gap

08 · Recommendations

How BrowserStack can close the gap

The 9-second overshoot is small enough that one or two well-placed BS-side optimizations clear it. Ranked by likely impact.

High impact

1 — Reduce firecmd_time outliers

Median firecmd_ms is 8.6 s, P90 is 10.6 s. But 6 sessions in our dataset hit firecmd_ms > 90,000 ms (one as bad as 188 s) — clearly a separate retry/contention path. Without those outliers, cloud P90 drops to ~749 s — inside the 1.1× ceiling.

What this is about: "firecmd" is the time between the customer triggering a test and Maestro actually starting to do anything on the phone. Think of it as the wait at a coffee shop between placing your order and the barista picking it up. Most of the time it's quick (about 8.6 seconds), but a handful of sessions waited over a minute and a half — far longer than they should have. Those few stragglers are what push our 90th-percentile number over the line.

Why it matters: if BrowserStack just fixes those rare slow starts, the cloud number drops below the 1.1× target with no other changes. This is the single highest-leverage fix.

Fix: tighter timeouts on registration handshake, warmer device pools to skip full driver bootstrap on every session.

High impact

2 — Pre-warm the Maestro test runner on the device

On local, we start the XCTest runner once and reuse it for 10 consecutive sessions — driver_setup is just 711 ms P90. Cloud sets up the runner per session. test_install_time P90 is 2.0 s on cloud — pure repeated work.

What this is about: Maestro needs a small "helper app" running on the iPhone that does the actual tapping and typing. Installing this helper takes about 2 seconds. On local, we install it once and reuse it across 10 back-to-back sessions. On cloud, BrowserStack reinstalls it for every single session — like a barista re-grinding their beans for every cup instead of using the same grinder all morning.

Why it matters: a customer running 100 benchmark sessions pays this 2-second tax 100 times — that's over 3 minutes of pure waste, with no benefit. Pre-warming the runner once per device per "queue" would give that time back.

Fix: Keep test bundles resident on a "Maestro-warm" device for N consecutive sessions in a customer's queue.

Medium impact

3 — Surface per-step Maestro timings in BigQuery

Right now BS only exposes session-level phases. Customers can't see whether the cloud iPhone is genuinely slower per rep, or whether a specific step (assertVisible waiting longer for the list to settle, etc.) is the bottleneck.

What this is about: Maestro keeps a detailed log of how long every individual tap, type, and assertion took. Locally, we used those logs to break down our session into rich detail. On cloud, BrowserStack throws those logs away after the test ends — they only keep the summary numbers (total time, install time, etc.). It's like getting a restaurant bill that says "$80 total" with no itemized list.

Why it matters: when a customer's cloud test is slower than expected, they have no way to tell where the slowness is coming from. Was it a slow tap? A flaky assertion that retried? They can't know — and so they can't even ask BrowserStack the right diagnostic question. This isn't a performance fix; it's a visibility fix that turns vague complaints into actionable bug reports.

Fix: Parse the Maestro debug log BS already collects on every session and surface step-level timings in the existing BQ table. Diagnostic-only feature; no perf impact.

Medium impact

4 — Reduce flake rate (10/100 = 10%)

Failed sessions cluster at 40–228 s with abnormally high firecmd (16–188 s) and elevated test_install_time. These look like infra-side retries that exhausted patience. Even though they don't affect P90, they reduce sample size and create noise for customers running shorter benchmarks.

What this is about: 10 out of 100 sessions in our cloud run failed early — not because the test itself broke, but because BrowserStack's own setup hit some delay or retry that timed out. Looking at the numbers, every failed session had unusually slow startup phases, suggesting they were stuck in infrastructure-level retries that eventually gave up.

Why it matters: a customer running a small batch (say 5 sessions) might lose half of them to flake and not realize it's BrowserStack's plumbing rather than their own test. Even a customer running 100 sessions sees noise that makes it harder to spot real regressions in their app.

Fix: root-cause the 90+ s firecmd contentions; surface a "retry succeeded" marker so flakes are distinguishable from real failures.

Low impact

5 — Document the upstream Maestro real-device limitation

Maestro 2.5.0's auto-spawn iOS-real-device path is broken upstream (NotImplementedError on launchApp). The working workflow requires PR #2856 + manual xcodebuild test-without-building + --port. We rediscovered this from scratch.

What this is about: as detailed in the Journey section, the official Maestro release literally cannot launch an app on a brand-new iPhone — the code is missing. We had to build a community-patched version ourselves before any local benchmarking could happen. There's no warning about this anywhere in the official docs; we burned hours rediscovering the workaround from scratch.

Why it matters: any BrowserStack customer who wants to compare their cloud results against a local baseline (which is exactly the kind of comparison this report does) will hit the same wall and waste the same time. A short documentation page would save every future user that detour.

Fix: BS docs should call out this limitation and link the working setup.

Low impact

6 — Add an "iOS reduced-overhead" benchmark mode

For benchmarking workloads, BS could skip re-signing on identical app uploads (upload once, BS re-signs once, all 100 sessions reuse the resigned blob).

What this is about: every iOS app uploaded to BrowserStack gets re-signed with BS's own developer certificate so it can run on their managed devices. Today this re-signing happens repeatedly even when the same app is being run dozens of times in a row. It's like a security checkpoint re-scanning the same suitcase every time it goes through, even though nothing has changed.

Why it matters: for benchmark and regression-testing workloads — where the exact same app is run hundreds or thousands of times — this repeated work is pure overhead. An opt-in "I'm running the same app over and over" flag would let BS cache the signed binary and skip the redundant work.

Fix: Optional flag on the build trigger payload.

Bottom line

The per-rep work itself (~80 % of session_total) is hardware- and iOS-bound — neither BS nor we can speed up eraseText on iOS. The reachable budget is the pre-flight overhead. Items 1 + 2 alone should put cloud back inside the 1.1× ceiling without any change to the workload.

09 · Methodology

How we measured

For reproducibility and audit. Skip if you trust the numbers.

Local measurement stack
  • Host: macOS 25.4 / Xcode 26.4.1, free Personal Team developer cert.
  • Device: iPhone 17 (iOS 26.4.2), USB-tethered.
  • Maestro CLI: Custom build from PR #2856 (omnarayan fork) with TestCommand.kt:496 patch (port routing) and Swift runner fixes (0.0.0.0 bind, ScreenSizeHelper rewrite).
  • USB bridge: pymobiledevice3 usbmux forward 22087 22087 --udid <udid> (replaced iproxy due to 7-min keep-alive ceiling).
  • Test runner: Pre-built once via xcodebuild build-for-testing, kept resident across all 10 sessions.
  • Per-iter cycle: uninstall app → install app → maestro test → uninstall app.
  • Phase parsing: Maestro debug log parsed by parse_maestro_log.py for step-level timings; bash script wraps with now_ms wall-clock for phase boundaries.
Cloud measurement stack
  • Build trigger: One BrowserStack Maestro build with 100 device entries (iPhone 17-26, ap-south-1).
  • Concurrency: ~25 sessions running in parallel waves (BS plan parallel cap).
  • Status polling: BS REST API every 30 s until terminal.
  • Per-session metrics: Pulled from BigQuery (browserstack-production.app_automate.app_automate_test_sessions_partitioned) ~50 min after build completion.
  • Phase columns from BQ: firecmd_time, app_install_time, test_install_time, test_download_time, app_download_time, duration.
  • Per-step (eraseText / inputText / assertVisible) timings: NOT in BQ — Maestro debug log not currently surfaced.
Percentile method

Linear-interpolation percentile (numpy default). For N=1, all percentiles equal the observation. For repeating loop-body steps, N = sessions × reps_per_session = 1,150 on local.

Cloud aggregates include all 98 terminal sessions (88 passed + 9 failed + 1 error). Failed sessions cluster at 40–228 s — well below P50 — so they affect Min and lower percentiles only. P50–P100 are pure passed-session statistics either way.

Sample sizes & confidence
  • Local: 10 sessions, range 674–683 s. Spread of 8.9 s = 1.3 % of the median. Confidence high.
  • Cloud: 98 sessions (88 passed). Range of passed sessions: 700–830 s (130 s = 17.7 % of median). Higher variance is expected for cloud — different physical devices, network, contention.
  • Sample size for cloud P90: at N=88, the P90 estimate has ±~1 rank uncertainty. The 759 s figure is robust to that.
